## Resumo executivo (≤5 linhas)

Arquitetura do workspace bem estruturada com symlinks funcionais entre hub e pipeline/vault, mas com 3 artefatos quebrados (arquivos com aspas no nome), vazio crítico em purification.jsonl, CI fragmentado (apenas hub e Atlas têm workflows ativos), e 3 parsers independentes consumindo corpus-data.json sem contrato compartilhado. companion-data.json tem apenas 5 itens vs 165 esperados. Sem dependências circulares détectadas.

- **[C-02]** `companion-data.json` tem apenas **5 itens** (não 145 como reportado na Wave 1.1). O sync_companion.py existe mas nunca foi executado ou seu output não foi persistido. Afeta: todo o dashboard do iconografia-companion opera com dados vestigiais.

## Achados — MAJOR

- **[M-01]** CI coverage fragmentado: apenas `hub/` (validate.yml) e `pipelines/Atlas/` (python-package.yml + python-package-conda.yml) possuem workflows ativos. Repos sem CI: `pipelines/indexing`, `pipelines/iconography-ingest` (é symlink para hub mas não tem workflow próprio), `apps/iconografia-companion`, `apps/iconografia-space`, `apps/iconografia-db` (não existe), `labs/iurisvision`.

- **[M-02]** O `validate.yml` do hub **não executa testes pytest**, apenas valida schemas. Isso significa que os 21 arquivos de teste em `tests/` (argos/, tools/, training/) nunca são executados no CI. Cobertura real: ~0% do código Python.

- **[M-03]** 3 parsers independentes para `corpus-data.json` sem contrato de schema compartilhado:
  1. `tools/scripts/sync_companion.py` (lê direto com `json.load`)
  2. `tools/scripts/csv_to_records.py` (lê CSV derivado do corpus_dataset.csv)
  3. `apps/iconografia-space/app.py` (lê arquivo local ou do HF dataset)
  
  Nenhum usa um schema JSON formalizado para validação de estrutura.

- **[M-04]** O diretório `hub/mnemosyne-scout/ideas/` reportado como ausente na Wave 1.3 **existe e está ativo** com 4 arquivos de reserva criativa. Porém não há evidência de que seja referenciado por qualquer script, notebook, ou workflow — funciona como repositório isolado.

- **[M-05]** Schema `master-record.schema.json` existe em `tools/schemas/` com 95 linhas, mas não há evidência de que os consumidores de `corpus-data.json` o utilizem para validação. O contrato é implícito, não enforced.

## Achados — MINOR

- **[m-01]** Symlink `pipelines/Atlas` no hub aponta para `/Users/ana/Research/pipelines/Atlas` (real directory). Symlink `pipelines/indexing` igualmente. Esses dois são reverse-symlinks (hub → pipelines). Isso é correto pela arquitetura documentada, mas crea um padrão assimétrico: `iconocracy-ingest` é symlink em pipelines apontando para hub, enquanto `Atlas` e `indexing` são symlinks em hub apontando para pipelines.

- **[m-02]** `hub/iconocracy-corpus/` contém múltiplos diretórios com propósito não documentado em AGENTS.md: `New Folder/`, `misc/`, `random outputs/`, `Notas e Textos/`, `PHD/`, `Biblioteca - Uso simbólico/`. Esses não são mencionados como buckets ou categorias no workspace index.

- **[m-03]** `archive/` dentro do hub contém `llm-outputs/` e `notas-legacy/` com conteúdo aged. documentações mencionam que `archive/` é read-only e não deve ser referenciado por código ativo. Há referências a `/Users/ana/Research/archive/js-genai` via symlink `js-genai` no hub pointing to archive/... Verificar se esse symlink é realmente usado ou é legacy.

- **[m-04]** O workflow `deploy.yml` no hub referencia `requirements-iconocode-gemma4.txt` e `requirements.txt` para install, mas não há passo de validação de schema após deploy.

- **[m-05]** `pipelines/indexing/` não tem workflows. Seus subdiretórios `corpus-scout-agent/` (TypeScript) e `gallica-mcp-server/` são projetos Node.js separados, cada um com package.json mas sem GitHub Actions.
