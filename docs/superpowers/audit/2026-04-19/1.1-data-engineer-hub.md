# Audit Report — data-pipeline — agency-data-engineer
Escopo examinado: tools/scripts/, tools/schemas/, data/processed/records.jsonl, corpus/corpus-data.json, data/processed/purification.jsonl, companion-data.json, notebooks/, .github/workflows/
Arquivos analisados: 80+
Data: 2026-04-19

## Resumo executivo (≤5 linhas)

O pipeline records.jsonl → corpus-data.json → companion-data.json → HF releases possui contratos de schema bem definidos eatomic writes protege contra corrupções parciais, mas 11 registros em records.jsonl falham validação schema (valores 4 em indicadores 0–3), purification.jsonl está vazio (0 registros) e companion-data.json está stale (145 vs 165 itens reais). Não há testes para o ETL canônico nem cobertura CI para companion-data.json ou notebooks. A regressão silenciosa na qualidade dos dados é o maior risco atual.

## Achados — CRITICAL (bloqueante, fix antes de qualquer release)

- [C-01] **Schema validation falhando em 11 registros** · `data/processed/records.jsonl:linhas diversas` · 11 registros têm valores 4 nos indicadores `monocromatizacao`, `serialidade` ou `inscricao_estatal`, violatesdo o maximum: 3 do master-record.schema.json. O script validate_schemas.py corretamente rejeita (exit code 1, 154/165 válidos). O pipeline produz dados schema-invalid.地元は коррупции нет — são 11 registros com valores corrompidos vindos da planilha corpus_dataset.csv durante a migração csv_to_records.py. · Remediação: executar `python tools/scripts/code_purification.py --export-csv` para identificar IDs com valor 4; limpar ou corrigir manualmente os 11 registros em corpus_dataset.csv e reimportar via csv_to_records.py. ship primeiro com os 154 registros válidos enquanto a correção é preparada.

- [C-02] **purification.jsonl vazio quebra o gate de release HF** · `data/processed/purification.jsonl` · Arquivo existe com 0 registros. O script build_hf_release.py chama `validate_schemas.py` com `check=True` na purify-record schema que retorna exit 0 para 0/0 registros válidos (vazio ≡ válido por definição). O release é construído com purification.jsonl vazio, copiado para o snapshot HF como se fosse dados legítimos. Esto é uma silenciosa violação do contrato de dados. · Remediação: em build_hf_release.py, adicionar pré-condição `if len(purification) == 0: raise SystemExit('purification.jsonl está vazio — codifique itens antes de fazer release')`. ship antes de qualquer publish hf.

- [C-03] **companion-data.json stale — 145 vs 165 itens** · `companion-data.json:3` · O campo `corpus_total` diz 145 mas corpus-data.json e records.jsonl têm 165 registros. O sync_companion.py é gerador mas companion-data.json está tracked no git e não há CI para detectar dessincronização. O campo `coded: 0` é hardcodado para todos os países, indicando que sync_companion.py não está sendo executado após as codificações. · Remediação: adicionar `companion-data.json` aos paths de trigger do validate.yml; executá-lo no pre-release gate; considerar se companion-data.json deve ser gerado no momento do build e não commitado.

## Achados — MAJOR (degrada qualidade, deve corrigir no próximo ciclo)

- [M-01] **Sem testes para o ETL canônico** · `tests/` (86 testes totais, nenhum para csv_to_records.py, records_to_corpus.py ou build_hf_release.py) · 18 arquivos de teste cobrem ARGOS, ferramentas de ingest e-normalização, mas não o fluxo canonical records→corpus→HF. O pipeline de maior risco não tem regressão. · Remediação: adicionar testes em tests/tools/ para csv_to_records.py (idempotência, schema output), records_to_corpus.py (merge vs replace, contagem de itens) e build_hf_release.py (drift detection, arquivo ausente).

- [M-02] **companion-data.json fora do gate CI** · `.github/workflows/validate.yml:5-15` · Os triggers cobrem corpus/**, data/processed/**, tools/schemas/**, tools/scripts/** mas NÃO companion-data.json, notebooks/ ou o próprio workflow. Edições em companion-data.json são invisíveis para CI. · Remediação: adicionar companion-data.json aos triggers de validate.yml; considerar adicionar notebooks/ para testes de reprodutibilidade (executar nbconvertdry-run).

- [M-03] **Deprecation warning em RefResolver** · `tools/scripts/validate_schemas.py:16` · `from jsonschema import Draft202012Validator, FormatChecker, RefResolver` — RefResolver está deprecated desde jsonschema v4.18.0. Funciona atualmente mas quebra em futura atualização do pacote. · Remediação: migrar para a biblioteca referencing (https://github.com/python-jsonschema/referencing) como sugerido na warning; atualizar requirements.txt se necessário.

- [M-04] **validate.yml não lista a si mesmo como trigger** · `.github/workflows/validate.yml:5-15` · Editar o workflow de validação não dispara o próprio workflow. Um commit que enfraquece a validação passa despercebido. · Remediação: adicionar `.github/workflows/validate.yml` aos push.paths do workflow.

- [M-05] **Dual source de purification data sem reconciliação** · `data/processed/purification.jsonl` vs `records[].purificacao` (embedded) · purification.jsonl é artefato independente produzido por code_purification.py; os indicadores também vivem no bloco purificacao de cada registro em records.jsonl. build_hf_release.py consome ambos. Se os dados divergem (e.g., code_purification.py exporta CSV mas records foi migrado antes de certa correção), não há reconciliação automática. · Remediação: code_purification.py deve atualizar records.jsonl in-place, ou purification.jsonl deve ser a única fonte (records[] purificacao extraído do purification.jsonl). Não manter duas fontes sem reconciliação.

- [M-06] **8 registros com placeholder URL** · `data/processed/records.jsonl:8 registros` · 8 registros têm URL `https://iconocracy.corpus/placeholder/{item_id}` — são itens sem fonte pública. OAGENTS.md indica que todos os itens precisam existir no Google Drive + drive-manifest.json, mas não há validação automática para garantir que placeholders estão accounted no manifest. · Remediação: adicionar check em validate_schemas.py ou no pre-commit hook para detectar mais de N placeholders e alertar.

## Achados — MINOR (polimento)

- [m-01] **sync_companion.py hardcodes coded: 0** · `tools/scripts/sync_companion.py:111` · Todos os países mostram `coded: 0` permanentemente, mesmo após code_purification.py ter executado. O status de codificação real está nos registros (field purificacao existe em todos os 165 registros). O campo coded no companion-data.json não reflete o estado real. · Remediação: derived do count de registros com purificacao não-nula por país no records.jsonl.

- [m-02] **23 arquivos fig_*.png em data/processed/** · `data/processed/fig_*.png` · Figuras geradas por notebooks são commitadas no repositório. Isso viola o princípio de que artefatos gerados devem estar em output/ ou .gitignore. Porém, são usadas como assets para o build do manual PDF (docs/build_manual_pdf.py). · Remediação: mover fig_*.png para output/figures/ e ajustar caminhos nos notebooks e build scripts; adicionar output/ ao .gitignore exceto README.

- [m-03] **Notebooks assumem 145 itens fixos** · `notebooks/07_dimensionality.ipynb:13` · Comentário diz `(145 itens, 10 indicadores)`hardcoded. Com o corpus agora em 165, análises都是以 145为基础 mas não há validação. Os notebooks rodam sem erro aparente mas produzem resultados stale. · Remediação: adicionar cell de verificação no início de cada notebook: `assert len(df) >= 145, f'Esperado >=145, encontrado {len(df)}'`.

- [m-04] **records_to_corpus.py merge mode preserva itens órfãos** · `tools/scripts/records_to_corpus.py:217-229` · Itens em corpus-data.json que não aparecem em records.jsonl são mantidos como-és (linha 228: `entry = dict(item)`). Este é um comportamento intentional para permitir extensão progressiva, mas significa que rodar records_to_corpus.py --merge repetidamente com records.jsonl encolhendo não remove itens órfãos do corpus. · Remediação: documentar claramente este comportamento; adicionar flag `--prune-orphans` se remoção for desejada.

- [m-05] **batch_id fixo na migração** · `tools/scripts/csv_to_records.py:34` · `MIGRATION_BATCH_ID = '00000000-1c0c-4842-8a1a-iconocracy001'` é hardcoded. Todos os registros de migração têm o mesmo batch_id não-UUID. Isso é válido pelo schema ( matching the non-UUID pattern), mas impede rastreabilidade de quando cada registro foi migrado. · Remediação: se rastreabilidade por migração for importante, usar timestamp ou número sequencial no batch_id; se não, documentar que batch_id é apenas agrupamento lógico.

## Pontos fortes (o que NÃO mexer)

- **Atomic writes** em todos os scripts ETL (records_to_corpus.py, csv_to_records.py, code_purification.py) — использую tempfile + rename, garantindo que arquivo corrupto não seja sobreposto em caso de erro durante escrita.
- **Schema well-defined** com 7 schemas JSON Schema Draft 2020-12 cobrindo master-record, iconocode-output, webscout-output, purification-record, argos-manifest, webscout-input, research-cluster. additionalProperties: false em todos os schemas.
- **CI tem consistency check** entre records e corpus (linha 61-63 do validate.yml: `if [ $RECORDS -ne $CORPUS ]; then exit 1; fi`) — Isto capturaria desvio de contagem no gate.
- **records_to_corpus.py merge mode** — 设计 inteligente de preservar campos enriquecidos (panofsky, institution, etc.) do corpus existente enquanto atualiza apenas campos autoritativos do records.jsonl.
- **Deterministic item UUIDs** via uuid5 + fixed namespace — garante que o mesmo item_id sempre produza o mesmo UUID, eliminando duplicatas acidentais em re-migrações.
- **Validation script exits non-zero** quando há falha de schema — CI step falha corretamente quando dados estão quebrados.
- **Dry-run flags** em todos os ETL scripts — permite inspeção antes de escrita.
- **AGENTS.md** bem documentado com comandos de validação, fluxos e guardrails.
