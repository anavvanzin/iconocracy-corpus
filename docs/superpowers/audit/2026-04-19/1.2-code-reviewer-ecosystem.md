## Resumo executivo (≤5 linhas)

O ecossistema ICONOCRACY não apresenta vazamentos de secrets hardcoded nem dependências com vulnerabilidades críticas, e o código TypeScript é relativamente limpo (sem `any` nas APIs de rede, sem `@ts-ignore`, sem `console.log` em produção). Os principais riscos são: (1) uso intensivo de `as any` na camada Firebase do iurisvision criando risco de Runtime TypeError silencioso; (2) ausência total de testes em 5 dos 7 repos do ecossistema, com o hub concentrando 36 testes focados em ARGOS mas cobrindo nenhum dos pontos mais críticos — APIs do Cloudflare Worker, parsers de corpus-data.json, e pipeline de sync da tese; (3) fragmentação lógica entre os leitores de corpus-data.json (hub records.jsonl → KV → Companion Worker vs. leitura direta pelo corpus-scout-agent).

## Achados — MAJOR (degrada qualidade, deve corrigir no próximo ciclo)
- [M-01] `as any` prevalente em iurisvision · `labs/iurisvision/src/lib/firebase.ts:12,18` + 9 ocorrências em componentes · O Proxy Firebase e acessos a Firestore QuerySnapshot usam `as any` sistematicamente, eliminando a verificação de tipos exatamente na camada mais crítica (autenticação + persistência). Erros de propriedade inexistente serão Runtime Exception, não TypeScript error. · Refatorar para interfaces tipadas (`FirestoreDoc`, `UserProfile`) e usar `getDoc` com tipagem explícita. Mesmo padrão presente em `ConnectionMap.tsx`, `ThesisDashboard.tsx`, `CorpusExplorer.tsx`, `KnowledgeBase.tsx`.

O caller em UI não sabe se o PUT falhou, criando possível perda silenciosa de dados do diário e da radiografia. · Retornar `{ok: false, error}` ou usar um toast de erro.
- [M-03] corpus-data.json leído em 3 pontos sem canonical parser compartilhado · `pipelines/indexing/corpus-scout-agent/src/index.ts:49` (leitura direta via `fs.readFileSync`), `apps/iconocracia-companion/scripts/sync-thesis-content.mjs:76` (leitura via `fs.readFile` + `readJsonMaybeList`), `apps/iconocracia-companion/src/app/main.jsx:10` (fetch API → `r.json()` fallback chain) · O corpus-scout-agent valida IDs contra o JSON mas não passa pelo mesmo código de parsing que o sync script. Divergências de schema entre o JSON do hub e o JSON do KV (compactado pelo worker) não serão detectadas. · Extrair parser canônico para `shared/corpus-parser.ts` e usar em todos os três pontos.
- [M-04] Nenhum teste em 5 dos 7 repos · `gallica-mcp-server/`, `iconocracia-companion/`, `iurisvision/`, `corpus-scout-agent/`, `Atlas/` · Apenas o hub (iconocracy-corpus) tem testes (36). Os demais são zero. Release de qualquer superfície sem cobertura de regressão é risco. · Ver seção Recomendações para priorities de teste por superfície.

## Pontos fortes (o que NÃO mexer)

- **Zero secrets hardcoded**: não há API keys, tokens, ou credenciais de Firebase/Gallica/Anthropic em nenhum src/. Todos os accessos são via `process.env.VAR` ou env do Cloudflare Workers (CF_ACCESS_AUD, CF_ACCESS_TEAM_DOMAIN). Pattern correto.
- **Nenhuma ocorrência de `@ts-ignore` ou `@ts-expect-error` em toda a codebase TypeScript** — o que é notável para um projeto com 6 meses de desenvolvimento.
- **Nenhum `console.log`留下来** em nenhum src/ de produção — todos os logs de debug estão restritos a scripts de build (`build.mjs:41`) e ao entrypoint do corpus-scout-agent. Comportamento correto.
- **Sem empty catch blocks irônicos**: os catch vazios em `main.jsx:11,14,24,44,55,62,69` são todos fallbacks deliberados que retornam `[]` ou `{}` — padrão idiomático para JS/React.
- **Atlas corpus.py** é código Python maduro: usa dataclasses, tipagem em campos, validação com `validar()`, e vocabulário controlado (PAISES, FIGURAS, SUPORTES, REGIMES) documentado com docstrings extensivas e referências acadêmicas. Não alterar a estrutura.
- **gallica-mcp-server** é código bem estruturado: separa concerns em `services/` (iiif, oai, sru) e `tools/` (search, metadata, image), usa Zod para validação, transport é abstrato (stdio vs HTTP), e não tem dependência de credenciais (Gallica é open-access). Bom projeto.
- **iconocracia-space/app.py** usa `hf_hub_download` corretamente para leitura do dataset público, sem hardcoded tokens. API pública do HuggingFace.

## Dependências inter-domínio (se achado afeta outro agente)

- [M-03] `sync-thesis-content.mjs` lê corpus-data.json do hub e valida relatedCorpusIds. Qualquer mudança no schema do corpus-data.json (adicionar/remover campo `id`) vai quebrar primeiro o sync script e depois todos os outros consumidores. Depende de `agency-data-engineer` para schema governance.
- [M-01] O Firebase Proxy pattern em `firebase.ts` depende de `@shared` (workspace shared package) para `getDb()` e `getAuthInstance()`. Se `@shared` não for publicado/atualizado, o iurisvision quebra em Runtime com erro de module resolution silencioso. Depende de `agency-frontend-developer` para manutenção do `@shared` package.
- O corpus-scout-agent (TypeScript) orquestra o gallica-mcp-server via `mcp__gallica__*` tools. Se a API SRU/OAI do Gallica mudar de formato de resposta, o corpus-scout-agent não tem fallback — apenas retry loop com 2 attempts. Depende de `agency-outbound-strategist` para manutenção do fallback pattern.

## Recomendações priorizadas (top 5)

1. **Fix `window as any` + substituir Firestore `as any` casts** em iurisvision por interfaces tipadas — risco de Runtime TypeError silencioso em produção Firebase.
2. **Criar testes para corpus-scout-agent** (critical path: leitura de corpus-data.json, validação de duplicatas, HF sync). Ponto de maior risco de corrupção de dados do corpus.
3. **Criar testes para gallica-mcp-server** (critical path: parsing SRU XML, construção de URLs IIIF, validação de license rights). Ponto de entrada de dados do acervo BnF.
4. **Unificar parser de corpus-data.json** em shared utility — elimina o risco de divergência entre o hub (source of truth), o KV do Cloudflare Worker, e o corpus-scout-agent.
5. **Adicionar testes para sync-thesis-content.mjs** (critical path: parsing do manifest.yaml, geração de embeddings, write de index.json/search.json). Queda do sync quebra todo o search da tese.
